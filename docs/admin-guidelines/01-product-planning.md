# 기획 지침 (Product Manager)

## 어드민 목적과 사용자

**목적:** 운영팀이 Lucky Piggy 서비스의 핵심 루프(미션 → 티켓 → 추첨 → 당첨)를 안정적으로 운영하고, 이상 징후를 빠르게 감지·대응하기 위한 내부 도구.

**사용자별 역할:**
| 역할 | 주 목적 |
|------|---------|
| 운영 매니저 | 추첨 진행, 당첨자 처리, CS 대응 (주 사용자) |
| 마케터 | 미션/이벤트 설정, 쿠폰 발급, 사전등록 관리 |
| 개발자/기술 PM | 로그 확인, 데이터 무결성 점검, 긴급 이슈 대응 |
| CEO/경영진 | KPI 대시보드 열람 (읽기 전용) |

---

## 우선순위별 핵심 기능

### Must Have (출시 전 필수)
- 사용자 조회·상태 변경 (정지/해제)
- 티켓 발급 내역 조회 및 수동 지급/차감
- 주간 추첨 실행 및 당첨자 확정·발표
- 상품권/포인트 지급 처리 및 상태 추적
- KPI 대시보드 (DAU/MAU, 티켓 발급량, 추첨 참여율)

### Should Have (초기 운영 안정화)
- 미션별 참여 현황 및 ON/OFF 토글
- 쿠폰 코드 생성·배포·사용 현황
- 부정 사용자 탐지 리포트 (비정상 티켓 패턴)
- 푸시 알림 발송 (당첨 발표, 이벤트 공지)

### Nice to Have (성숙 단계)
- A/B 테스트 설정 (미션 노출 순서, 보상량)
- 코호트 분석 (가입일별 잔존율)
- 룰렛/보물캐기 파라미터 실시간 조정

## KPI 대시보드 지표

- **사용자:** DAU / MAU / 신규 가입 / 이탈률
- **티켓:** 일별 발급량 / 미션별 발급 비율 / 평균 보유량
- **추첨:** 회차별 응모자 수 / 응모율(DAU 대비) / 당첨 처리 완료율
- **미션:** 미션별 완료율 / 광고 시청 수 / 만보기 인증 수
- **수익:** 광고 수익 추정 / 쿠폰 사용율 / 지급 비용 합계
- **건강 지표:** 부정 행위 의심 계정 수 / 오류 발생 건수

## 운영 시나리오

### 당첨 발표 프로세스
1. 추첨 마감 확인 → 응모자 집계 → 추첨 실행
2. 당첨자 명단 검토 (부정 사용자 필터) → 확정 → 푸시 발송
3. 상품권/포인트 지급 → 완료 상태 업데이트 → 미지급 건 재처리

### 테스트 컨트롤 패널 (당첨자 관리 상단)

cron 스케줄 없이 추첨 프로세스 전 단계를 수동으로 시뮬레이션하는 패널.
스테이징/개발 환경에서 QA 용도로 사용. 프로덕션에서도 표시되나 강한 경고 배너 노출.

| 버튼 | 활성 조건 | 실행 내용 |
|------|-----------|-----------|
| [Step 1] 응모 마감 | status = active | draws.status → closed (end_date 무시) |
| [Step 2] 당첨자 판정 | status = closed | winning_numbers/bonus 저장 → judge_draw_winners RPC → status → drawn |
| [Step 3] 당첨자 발표 | status = drawn | status → completed, 다음 회차 활성화 (푸시 생략) |
| 초기화 | status ≠ completed | draw_winners 삭제 + draw_entries 롤백 + status → active |

**환경 표시:** 헤더에 STAGING / ⚠️ PRODUCTION / LOCAL·DEV 배지 노출.
**관련 파일:**
- `src/app/(admin)/draws/_components/TestControlPanel.tsx`
- `src/app/(admin)/draws/actions.ts` — closeDrawAction, judgeWinnersAction, publishDrawAction, resetDrawAction
- `src/lib/supabase/draws.ts` — closeDrawForTest, judgeWinnersForTest, publishDrawForTest, resetDrawForTest

### 쿠폰 발급 프로세스
1. 쿠폰 조건 입력 (금액, 수량, 만료일) → 코드 생성
2. 배포 채널 설정 → 사용 현황 모니터링

### CS 이슈 대응
- 티켓 이상 급증 → 해당 미션 즉시 OFF → 원인 조사 → 티켓 롤백
- CS 문의 → 사용자 ID 검색 → 티켓·추첨 이력 확인 → 수동 보정

## 기획 원칙

1. **Undo 가능한 액션 설계** — 티켓 지급·차감, 당첨자 확정 전 미리보기·확인 단계 삽입
2. **권한 분리** — 역할별 접근 범위 명확히 (운영자/매니저/개발자/경영진)
3. **감사 로그 필수** — 모든 데이터 변경 액션은 누가, 언제, 무엇을 변경했는지 기록
4. **모바일 어드민 불필요** — 데스크탑 해상도 최적화, 모바일 우선순위 낮음
5. **실시간보다 정확성 우선** — 추첨·지급 기능은 처리 속도보다 데이터 정합성이 중요
6. **운영자 실수 방지 UX** — 위험한 액션(정지, 롤백, 대량 지급)은 2단계 확인 필수
7. **KPI는 맥락과 함께** — 숫자 단독 표시 지양, 전주/전월 대비 변화율 병기
